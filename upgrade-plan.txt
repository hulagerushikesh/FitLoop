Phase 1: Design System & Visual Overhaul

Goal: stop the app from looking like a prototype. This is the single highest-leverage phase — it's what makes everything after it feel professional even before new features land.

1. Create a centralized design system: `theme/` folder with:
   - colors.ts (semantic tokens: background, surface, surfaceElevated,
     textPrimary, textSecondary, border, accent, success, warning,
     danger — plus full light AND dark mode palettes)
   - typography.ts (a type scale: display, h1, h2, h3, body, bodySmall,
     caption, label — using a distinctive font pairing, not system
     default. Suggest Inter or Manrope for body text and a slightly
     heavier weight for numbers/stats since this is a data-dense app)
   - spacing.ts (4/8/12/16/24/32/48 scale — no magic numbers in
     component styles after this)
   - radii.ts and shadows.ts (consistent corner radius and elevation
     across cards/buttons/modals)

2. Build a shared component library in `components/ui/`:
   - Button (primary/secondary/ghost/destructive variants, loading state,
     icon support, haptic feedback on press via expo-haptics)

     for macro/calorie targets — replace any flat progress bars with
     these where it's the "hero" metric on a screen)
   - Input, NumberInput (with +/- steppers for reps/weight — much faster
     than typing on a phone mid-set)
   - Badge, Chip (for muscle group tags, PR badges, streak flags)

   - Skeleton loaders (shimmer placeholders) — replace any bare spinners

   - Toast/Snackbar for success/error feedback instead of Alert.alert
     where appropriate

3. Add a theme context + toggle: system/light/dark, persisted locally.
   Migrate all existing screens to consume the theme instead of hardcoded
   colors — grep the codebase for hex codes and inline color strings and
   replace them.

4. Install lucide-react-native for iconography if not already present.
   Replace any emoji-as-icon or missing-icon spots across the app.

5. Add subtle motion: screen transitions, card press states (scale down
   slightly on press), number count-up animations for stats (e.g., when
   the calorie target loads, animate from 0 to the value). Use

Phase 3 — Workout Module: Speed, Depth, and Intelligence

This is the phase where we close the gap with Strong (logging speed) and Fitbod (adaptive intelligence).
1. LOGGING SPEED (Strong-style):
   - Auto-populate each set's weight/reps with the value from the user's
     last session of that exact exercise, shown as a subtle placeholder/
     pre-fill they can tap to accept or edit.
   - Replace text-input-only set entry with NumberInput steppers (+/-)
     sized for one-thumb use, plus direct tap-to-type for exact values.
   - Add a "repeat last set" quick-add button for supersets/dropsets of
     the same movement.
   - Add a plate calculator (given total weight + bar weight, show which
     plates go on each side) as a small utility accessible from the set
     logging screen.
   - Add a 1RM estimate (Epley formula: weight × (1 + reps/30)) displayed
     live as the user logs a heavy set, and a dedicated "PR" celebratory
     toast/animation when a logged set beats their historical best for
     that exercise.

2. SUPERSETS & ADVANCED SET TYPES:
   - Extend workout_exercises / workout_logs schema (new migration file)
     to support: superset grouping (link 2+ exercises performed back to
     back), drop sets (multiple weight/rep pairs within one logical set),
     and to-failure flag (AMRAP sets).
   - Update the routine builder UI to let users group exercises into a
     superset when building a routine.

3. MUSCLE RECOVERY MODEL (Fitbod-style, this is the AI differentiator):
   - New table: muscle_group_fatigue (user_id, muscle_group, last_trained_at,
     estimated_recovery_hours, rolling_volume_7d).
   - After each workout session is completed, an edge function or client-
     side job updates estimated recovery per muscle group hit, based on
     total volume (sets × reps × weight) and a recovery-hours heuristic
     per muscle group (e.g., large muscle groups like legs/back: ~72h,
     smaller groups like arms/core: ~48h — make these configurable
     constants, not magic numbers).
   - Surface this on the Home screen: a simple recovery indicator per
     muscle group (fresh / recovering / fatigued) using color-coded chips,
     and use it to suggest which of the user's existing routines is the
     best fit for "today" (don't build a full AI generator yet — start
     by ranking the user's own saved routines by how well they match
     currently-recovered muscle groups).

4. VOICE LOGGING (stretch goal, Vora-style):
   - Add a mic button on the set-logging screen using expo-speech (or a
     lightweight on-device speech-to-text lib compatible with Expo) that
     parses simple patterns like "135 for 8" or "60 kilos 10 reps" into
     weight+reps and pre-fills the input for confirmation (always require
     a tap to confirm, never auto-submit from voice alone — avoid mis-
     logs from parsing errors).

5. EXERCISE LIBRARY DEPTH:
   - Add short looping form-demo GIFs or embedded YouTube thumbnails
     (you already have YouTube deep-links — upgrade to show a thumbnail
     preview inline using the video ID, still deep-linking out to watch,
     no extra video hosting needed).
   - Add a custom exercise creation flow (name, muscle group, category,
     optional photo via Storage) so advanced users aren't limited to the
     55 seeded exercises.
   - Add exercise-level history: tapping any exercise from the library
     shows a mini chart of weight/volume progression over time for that
     specific movement (this doubles as a preview of Phase 6's analytics).

Acceptance: logging a full workout is noticeably faster than before
(fewer taps per set), superset routines can be built and logged, the

exercise shows a historical progress mini-chart.
